## 3. Root-cause findings

### 3a. execute-alerts.ts hard-crashed for ~1 hour — the day's biggest incident

`execute-alerts.ts` 500'd on **every single invocation from 10:13am to ~11:00am ET** (confirmed via Vercel logs, reproduced directly against the DB). Root cause: disabled signal types (ORB/IV/STTF) keep getting detected and inserted into `profit_targets` forever — `scan-confluence.ts` has no concept of the live-trading disable flag — and nothing ever claims or closes those legs since they never get real positions. IV alone (~3 legs/min all session) had piled up to **700+ open legs**, and the dedup query's `.in(profit_target_id, ids)` list serialized to a ~26,000-character request that Supabase rejected outright with a bare `Bad Request`.

Cron-job.org auto-disabled both `execute-alerts` and `track-profit-targets` after the repeated failures (the latter was almost certainly timing out against its own client-side limit, re-scanning the same growing backlog every run — the same class of issue already flagged in that file's comments from 2026-07-16).

**Fixed live**: bulk-expired the 658-leg stale backlog directly in the DB, patched `execute-alerts.ts` to filter disabled types out *before* building the dedup id list (commit `95e7de5`), user manually re-enabled both cron jobs. Confirmed healthy afterward — 11/11 clean runs on both.

**Not fixed, flagged for later**: `track-profit-targets.ts`'s `checkExpiry` only closes legs from a *prior calendar day* — same-day backlog growth (dominated by IV firing every ~60s) can still recur within a single session. Today's incident was the backlog crossing a breaking threshold; nothing stops it from happening again before tomorrow's boundary reset.

### 3b. The IWM 294C "winner" was a broken-safety-net fluke, not a good trade

Tier limit orders (10/15/20/25/30% scale-out ladder) were **never placed for any of today's 8 entries** — confirmed via Alpaca's order history: zero tier orders exist anywhere, all day, only the entry and stop orders. The last IWM position ran from $0.40 to over $1.30 (+225%) with no automated profit-taking active at any point — it should have been scaling out starting at $0.44.

While investigating live, found and fixed a related dead end: `notifyManualReview`'s failure handler only logs the *Firebase* error, never the original message it was trying to send (see 3c) — meaning every tier-placement failure's actual diagnostic text was silently lost, not just undelivered.

Attempted a manual sell-to-close and hit a separate, still-unexplained Alpaca rejection (`"account not eligible to trade uncovered option contracts"`) on market, limit, *and* stop orders, despite the account correctly showing `qty_available: 5` and options level 3 approved. The position was briefly left with **no protection at all** (I canceled the one working stop while troubleshooting). It ultimately closed via `monitor-executions.ts`'s own fallback tier-sell logic firing successfully once whatever was blocking sells cleared on its own — filled all 5 contracts at ~$1.30. **This was luck, not the system working** — a reversal during that unprotected window would have given back the gain instead.

Root cause of the tier-placement failures themselves is still not fully diagnosed.

### 3c. Push notifications were completely dead all day

Firebase service account credential was invalid (`invalid_grant: account not found`) — every `notifyManualReview` call today (tier failures, the execute-alerts outage, everything) fired into the void with zero visibility. User generated a fresh service account key from the Firebase console; replaced the Vercel env var and redeployed. Fix appears to have held (zero `invalid_grant` errors across 40+ subsequent cron invocations, versus constant errors before), but **a real successful notification send has not yet been directly observed** — nothing has needed to notify since the fix.
